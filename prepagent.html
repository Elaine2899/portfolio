PrepAgent - Telegram AI 備考助教與 RAG 知識庫

技術棧：Supabase Edge Functions (Deno/TypeScript) · Supabase Postgres + pgvector · Python · Google Gemini · Telegram Bot API · n8n（第一版）

專案性質：個人獨立開發｜全雲端架構、全天候運行

動機與問題定義

這是一個從自身需求長出來的專案：我準備報考研究所，統計學是必考科目，而我的主力教材是交通大學唐麗英老師的 YouTube 統計學課程——內容紮實，但影片教材沒辦法「問問題」，也不利於零碎時間的反覆複習。與其只當觀眾，我決定把它做成一個隨時可對話的備考助教，一次達成三個目的：實練 RAG 與向量資料庫技術、解決自己的備考需求、驗證 n8n 雲端工作流的可行性。

與 FunCtrl 的定位不同：FunCtrl 是團隊畢業專案，解決「在開放影片庫中定位知識點」的搜尋問題；PrepAgent 是我獨立開發的封閉語料問答系統——語料固定（一門課 29 支影片）、深度優先，目標是把一門課變成可以對話的知識庫。技術上也有一條延續線：FunCtrl 來不及嘗試的語音轉文字，在這裡成為第一步——29 支影片全部以 Whisper 轉出逐字稿，才有後面的一切。

備考的性質決定了兩個硬性約束：備考是長期戰，服務必須全天候在線、不能依賴自己的電腦開機——所以整個系統必須全雲端；考研不只考統計，所以架構必須模組化，讓新科目低成本接入（擴增已在進行中）。

方法選型與架構演進

初期選型的邏輯是「把力氣花在想練的地方」：

  • n8n Cloud：目的之一本來就是驗證 low-code 工作流，用視覺化節點把「Telegram webhook → 檢索 → 組 Prompt → Gemini 生成」整條流程接起來，快速得到能用的版本。
  • Telegram：做 bot 的門檻最低（官方 Bot API 免費、文件完整），而練習的主軸不在機器人本身——聊天介面的成本壓到最低，力氣留給 RAG。
  • Supabase（Postgres + pgvector）：向量庫、對話記憶、科目設定同住一個託管資料庫，免自架、免費額度夠用。

架構演進：n8n 到期，自寫後端接手。n8n Cloud 免費試用期滿後，我把整條流程翻寫成 Supabase Edge Function——telegram-bot 一支 function 處理 webhook、路由、RAG 與 Gemini 呼叫。已驗證的 RAG 流程不重做，只動兩處：更換查詢 embedding 模型、system prompt 從寫死「統計學／唐麗英老師」改為由科目設定驅動（為多科目擴增鋪路）。這次遷移等於同一個系統用 low-code 與手寫各做了一次：n8n 起步快，但方案時限是真實成本；自寫版本換來完全的控制權與零月費。

系統設計

全雲端架構

  • 運算：一支 Supabase Edge Function（Deno runtime、TypeScript）承擔全部服務邏輯——Telegram webhook 入口、指令路由、RAG 檢索、Gemini 呼叫。
  • 資料：同一個 Supabase Postgres 裡三張表各司其職——documents（pgvector 向量庫）、chat_memory（對話記憶）、chat_settings（科目設定）。
  • 保活：用 Supabase 內建的 pg_cron + pg_net 每天自我 ping 一次，防止免費方案因閒置而暫停服務。
  • 外部依賴只有兩個：Telegram（訊息入口，webhook 打進 Edge Function）與 Google Gemini API（LLM 生成＋嵌入，直呼 REST）。

本機只保留一支 Python harness 做一次性批次工作——Whisper 逐字稿的清洗、分塊、嵌入入庫（ingestion）與檢索評估——不是常駐服務。筆電關機，服務照跑：這正是動機一節「全天候在線」約束的落地。

三個關鍵設計決策

  • 直呼 REST、不掛 SDK——兩個執行環境同一原則：Google 官方 SDK 在 AI Studio API key 下預設綁定 v1beta，常因套件版本限制產生呼叫衝突。因此 Python ingestion 端（ingest.py、eval_retrieval.py）用 requests、Edge Function 端（index.ts）用原生 fetch，一律直接呼叫 Gemini REST 端點——依賴更少、兩端行為一致可控。
  • halfvec 解維度瓶頸：gemini-embedding-001 輸出 3072 維，超過 pgvector 標準 vector 型別 ivfflat 索引的 2000 維上限。改用 halfvec（16-bit 浮點）後索引上限升至 4000 維，順利建索引，還省下 50% 儲存與記憶體——精度損失在備考問答場景可忽略。
  • 滑動視窗分塊：Whisper 逐字稿先由 chunker.py 清洗剝離「行號 + tab」前綴，再以「20 行一塊、5 行重疊」切塊，保留段落間的語境聯繫，防止檢索時知識點斷裂。

對話記憶與出處

Bot 以 chat_memory 維持上下文：使用者先問「第二講教什麼」，之後只需輸入 /quiz，bot 就自動針對第二講出題，不需重新指定講次。回答一律附出處講次，抽查驗證大多正確——RAG 最常被詬病的「無憑無據」，用出處把可信度錨住。

檢索品質評估

這是從 FunCtrl 帶過來的教訓——先建評估體系、再談優化；獨立開發沒有隊友把關品質，這條原則更重要。評估由本機 harness 的 eval_retrieval.py 執行，可重複跑：

  • 評估集：16 題涵蓋基礎與進階課程的統計學問題，每題的正解講次（ground truth）以逐字稿關鍵字掃描建立。
  • 評估路徑與線上 bot 完全相同（RETRIEVAL_QUERY embedding → match_documents）——量到的就是使用者實際體驗到的檢索品質，不是實驗室數字。
  • 結果：Hit Rate@5 = 75%（12/16）；命中的 12 題中有 11 題正解排名第 1。

比 75% 更重要的是弄清楚那 4 題為什麼 miss。隔離測試顯示：4 題（ANOVA、標準差／變異數、集中趨勢、離散／連續隨機變數）的正解講次內最佳相似度僅 0.61–0.65，低於錯誤講次的 0.65–0.76——是真實語意落差，不是排名差一點。歸因有二：基礎講次的逐字稿高度口語化、以算例操作為主，與教科書式的定義型問句語意距離遠；ANOVA 查詢則被回歸講次中的 ANOVA 表內容吸走，屬合理混淆。對應的改進方向放在下一節。

限制與未來工作

  • 檢索的真實語意落差：miss 分析指出的口語化逐字稿 vs 定義型問句問題，對應三個已知解法——hybrid search（關鍵字＋向量）、contextual chunking（chunk 前綴加講次主題）、query rewriting——尚未實作，是下一輪迭代的主軸。
  • 公式的轉寫失真：Whisper 無法把口述數學式轉成 LaTeX，公式密集段落的逐字稿可讀性差，也直接影響這些 chunk 的嵌入品質——這是統計學這類數學重的科目特有的語料天花板。
  • 評估集規模：16 題、ground truth 以關鍵字掃描建立，覆蓋面與標注方式都有偏誤空間；擴大題庫、人工複核 ground truth 是評估體系本身的改進方向。
  • 語料單一：目前只有一門課、一位講者的口語風格，檢索參數與 prompt 是否過擬合這門課，要等第二個科目接入才會知道——多科目擴增已在進行中（chat_settings 架構就是為此準備的）。
  • 免費方案的脆弱性：pg_cron 自我 ping 能防閒置暫停，但免費額度的上限（Edge Function 執行時間、資料庫容量）終究是天花板；若使用者從一人變多人，成本模型要重新設計。

反思

這個專案讓我把 FunCtrl 學到的「評估先行」變成固定方法，而 miss 分析又往前推了一步——75% 這個數字本身不是重點，弄清楚「哪 4 題為什麼輸、輸在語意落差而不是排名」才是下一步迭代的地圖。另一個收穫是同一套系統用 low-code 和手寫各做一次的對照：n8n 適合快速驗證流程可行性，驗證完成後遷移到自寫程式的成本反而很低——因為流程已經被證明過了。「先用 low-code 畫地圖、再用程式碼鋪路」會是我之後 side project 的預設路徑。要說重來一次會改的：把過程記錄留好——n8n 階段的工作流沒截圖就退役了，這段演進現在只能用文字講。
