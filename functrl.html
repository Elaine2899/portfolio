FunCtrl - AI 智慧學習地圖平台

技術棧：FastAPI · Vue.js · ChromaDB · PostgreSQL · Gemini Pro · Docker · Railway

專案性質：畢業專案｜全國大專校院資訊應用服務創新競賽 (InnoServe) AI應用工具組第一名 🏆

動機與問題定義

題目源於指導教授對線上影片學習現況的觀察；團隊接手後先對既有工具做了一輪盤點，確認缺口真實存在：

  • YouTube：內建搜尋只比對標題與描述，檢索最小單位是「一部影片」，無法回答「這個概念出現在影片的哪一分鐘」；章節與時間戳需要創作者手動標注，覆蓋率與品質參差不齊。
  • Udemy、Hahow 等課程平台：提供結構化的課程編排，但內容封閉且需付費，無法涵蓋 YouTube 上大量的開放資源；站內搜尋同樣停留在課程與影片層級，做不到知識點定位。

我們設定的核心使用者是想自學新技能的任何人：面對海量影片資訊缺乏結構化引導、學習時間零碎、需要一條明確的學習路徑。同樣的缺口也存在於企業內訓場景——培訓影片缺乏系統化管理與搜尋、員工難以回顧重溫——這讓系統具備延伸為企業培訓方案的潛力。

因此我們把問題定義為：長影片語料上的細粒度語意檢索——輸入一句自然語言提問，系統要能定位到「哪部影片的哪個時間點」，並在這個檢索基礎上自動生成學習路徑。

方法選型與替代方案

技術路線的核心判斷是：在五人學生團隊的資源約束下，哪條路能把「知識點定位」的成效做到最好。我們比較過三條路：

  • 純關鍵字搜尋（倒排索引）：實作成本最低，但只能比對字面，處理不了「使用者用自己的話問、講者用另一種說法講」的語意落差——效果不會比 YouTube 內建搜尋好，做了等於沒做。
  • Fine-tune 專用模型：理論上限最高，但標注資料、訓練算力與反覆實驗的時間成本，超出畢業專案的資源規模。
  • 全套商用 API：整合最快，但按量計費對需要長期運行的服務不可負擔，檢索品質的調整空間也受限於供應商。

最終選擇開源 Embedding 模型的向量檢索，LLM 只在關鍵環節介入（查詢擴張、re-rank、摘要精煉）：向量檢索解決語意落差，開源模型把常態成本壓到可控，LLM 的花費集中在少數高價值步驟。這條路線也保留了迭代空間——後續評估階段能以 NDCG 為依據反覆更換 embedding 模型（詳見評估一節），而不需要動整體架構。

系統設計

整體架構

前端 Vue.js 部署於 Vercel；後端 FastAPI 承載 RAG pipeline（查詢嵌入 → 向量檢索 → 組合 Prompt → Gemini 2.5 Flash 生成回答），實作 30+ 支 RESTful APIs，涵蓋 JWT 認證、個人化推薦與測驗生成。資料層採雙資料庫設計：PostgreSQL 管結構化資料，ChromaDB 管向量——各用各的強項，避免單一資料庫兩頭將就。

部署起初在 Railway（平台自動容器化建置）；後期為節省 API 與伺服器維運成本，依指導教授的決策將服務遷移至地端實驗室主機。

三層檢索：標題 → 標籤 → 字幕

團隊討論後定下的分層設計：與其把整個字幕庫攤平成一池向量硬搜，不如由粗到細收斂——先以標題、標籤層鎖定候選影片，再進入字幕層做細粒度匹配，最後由 Gemini re-rank 並輸出精準時間點。分層有兩個實際考量：標題與標籤是短文本，第一層過濾處理快；且當不同影片涵蓋相似主題時，先鎖定影片再比對字幕，細粒度匹配不會被跨影片的相似內容干擾。

嵌入模型也按文本長度分工：短文本（標題、標籤）用 MiniLM-L6-v2，長文本（字幕）用 bge-m3。

三個關鍵設計決策

  • 正規化 + L2 距離取代餘弦相似度：比對向量時發現餘弦相似度運算較慢，我提出「先做長度正規化、再用 L2 距離檢索」——數學上與餘弦等價，但計算效能顯著提升，改善高併發下的查詢延遲。
  • Query Expansion：使用者的提問往往比講者的用語短而模糊，先用 Gemini 對原始 query 做語意擴張補足同義詞，再進檢索，提升 Recall。
  • 非同步 Self-Expanding 語料擴充：語料起步只覆蓋機器學習領域，為了應對領域外的查詢，以 Python asyncio 實作背景任務——當查詢信心值低於閾值，系統自動下載相關影片與字幕、完成嵌入與語意摘要，讓 ChromaDB 隨真實使用自動增量擴充。

檢索之上，系統再以 LLM 生成階層式學習地圖並持久化至資料庫，整合 AI 自動化測驗與學習進度儀表板，完成「檢索－學習－評量」的閉環。

評估方法與結果

檢索系統最難的不是做出來，是證明它排得好——「相關性排序是否合理」沒有現成的標準答案。我主導建立了一套評估體系：

  • 指標：引入 NDCG（Normalized Discounted Cumulative Gain），它同時懲罰「排錯順序」與「漏掉高相關結果」，比單純的準確率更貼合檢索場景。
  • 標準答案：由 3 位團隊成員獨立對 10 部目標影片進行 query 相關性標注，取平均以稀釋個人偏誤，作為理想排序（IDCG）的依據。
  • 迭代方法：計算系統實際得分與 IDCG 的偏離度，團隊據此反覆測試不同的開源 embedding 模型——選型靠數據，不靠感覺。

最終系統檢索得分率穩定在 88% 以上。

限制與未來工作

  • 語料覆蓋：測試語料集中在機器學習領域，跨出領域就沒有現成影片可檢索。Self-Expanding 背景下載（見系統設計）正是為此而生，但新語料的建立需要時間，做不到「即搜即得」。
  • 語言：時程內只完成英文檢索。雙語（中英）與多模態搜尋（畫面內容理解）是我們最想做而沒來得及做的方向。
  • 字幕依賴：僅支援本身提供 CC 字幕的影片。曾規劃以語音轉文字（ASR）為無字幕影片建立逐字稿，未及嘗試——這會直接決定系統能覆蓋多大比例的真實影片。
  • 評估規模：標注集僅 10 部影片，標注者為 3 位團隊成員而非獨立第三方，NDCG 分數的外部效度有限。

我的角色與反思

五人團隊的分工：兩位組員負責後端實作、一位負責前端，我負責全系統框架設計、頁面設計與前端製作，同時主持會議推動進度、製作對外簡報。後端雖非我實作，但兩個關鍵技術決策由我提出並推動落地：以正規化 + L2 距離取代餘弦相似度（見系統設計）、建立 NDCG 評估體系（見評估方法）。

反思：最值得的決定是先建評估體系、再談優化——沒有 NDCG 之前，檢索的好壞全憑感覺；有了客觀分數，每一次迭代才有方向。重來一次，我仍會照同樣的步驟走——這個順序被驗證是對的——差別是把每一步推得更快，讓時程尾端那些「來不及」的方向（ASR、雙語）有機會被納進來。
